목표 : 커머스 플랫폼의 핵심 도메인(브랜드, 상품, 좋아요, 주문)을 구축하여,
사용자가 브랜드별 상품을 탐색하고, 관심 상품에 좋아요를 남기며, 상품을 주문할 수 있는 전체 구매 흐름을 완성한다.

  ---

## 요구사항

### 브랜드 (Brand)

- 브랜드는 name(브랜드명), description(설명), logoUrl(로고 이미지 URL) 정보를 가진다
- 브랜드명은 중복될 수 없다
- 브랜드 목록을 조회할 수 있다 (페이징)
- 브랜드 상세 정보를 조회할 수 있다
- 관리자가 브랜드를 등록/수정/삭제(soft delete)할 수 있다

### 상품 (Product)

- 상품은 반드시 하나의 브랜드에 소속된다
- 상품은 name(상품명), description(설명), price(가격), stockQuantity(재고수량), status(판매상태) 정보를 가진다
- 상품 판매상태: SELLING(판매중), STOP_SELLING(판매중지), SOLD_OUT(품절)
- 재고가 0이 되면 상태가 자동으로 SOLD_OUT으로 변경된다
- 상품 목록을 조회할 수 있다 (페이징, 브랜드별 필터링)
- 상품 상세 정보를 조회할 수 있다 (브랜드 정보 포함)
- 관리자가 상품을 등록/수정/삭제(soft delete)할 수 있다
- 상품 가격은 0보다 커야 한다
- 재고수량은 0 이상이어야 한다

### 좋아요 (Like)

- 인증된 사용자가 상품에 좋아요를 누를 수 있다
- 이미 좋아요한 상품에 다시 좋아요를 누르면 좋아요가 취소된다 (토글 방식)
- 사용자별 좋아요한 상품 목록을 조회할 수 있다 (페이징)
- 상품 조회 시 좋아요 수가 함께 표시된다
- 삭제된 상품에는 좋아요를 누를 수 없다

### 주문 (Order)

- 인증된 사용자가 상품을 주문할 수 있다
- 한 번의 주문에 여러 상품을 담을 수 있다 (주문 항목별 수량 지정)
- 주문 시 각 상품의 재고가 주문 수량만큼 차감된다
- 재고가 부족하면 주문이 실패한다
- 주문 상태: ORDERED(주문완료), CANCELLED(주문취소)
- 사용자가 자신의 주문을 취소할 수 있다
- 주문 취소 시 차감된 재고가 복원된다
- 사용자별 주문 내역을 조회할 수 있다 (페이징)
- 주문 상세 정보를 조회할 수 있다 (주문 항목, 상품 정보 포함)

### 인증 (공통)

- 기존 회원 도메인의 인증 방식을 그대로 사용한다 (X-Loopers-LoginId, X-Loopers-LoginPw 헤더)
- 좋아요, 주문 기능은 인증된 사용자만 접근할 수 있다
- 관리자 기능(브랜드/상품 등록 등)에 대한 권한 분리가 필요하다

  ---

## 제안사항

- 기존 Hexagonal Architecture(interfaces → application → domain → infrastructure) 레이어를 동일하게 적용한다
- 기존 회원 도메인에서 확립한 패턴(Facade, Service, Repository 인터페이스 + Impl, BaseEntity soft delete)을 재사용한다
- 상품 재고 차감 시 동시성 문제를 고려한다 (낙관적 락 또는 비관적 락)
- 주문과 재고 차감이 하나의 트랜잭션 안에서 처리되어야 하는지, 분리 가능한지 검토한다
- 좋아요 수 집계 방식을 검토한다 (실시간 COUNT vs 비정규화 캐싱)
- 상품-브랜드 간 관계에서 브랜드 삭제 시 소속 상품 처리 정책을 결정한다
- API 버전은 V1으로 통일한다
